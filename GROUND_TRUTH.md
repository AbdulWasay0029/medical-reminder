## 3. Architecture (Three-Tier)

```
[Patient App]  [Guardian App]
     |               |
     React Native + Expo (TypeScript)
          |
     api.ts (Axios)
     JWT Bearer Token attached to every request via interceptor
          |
     FastAPI Backend (Python)
     Routes: /api/auth, /api/medicines, /api/dashboard, /api/logs, /api/guardian, /api/members
          |
     MongoDB Atlas
     Collections: users, medicines, logs, guardian_links
```

- The frontend is a **Logic-Lite client** — it displays data and triggers alarms. It does NOT compute statuses.
- The backend is the **Single Source of Truth** — it computes all dose statuses using server-side time logic.
- The database is **cloud-hosted** (MongoDB Atlas) so both Patient and Guardian see the same data in real time.

## 4. Authentication Flow

**File:** `backend/server.py`, `frontend/contexts/AuthContext.tsx`, `frontend/services/api.ts`

1. User registers via `POST /api/auth/register` sending: `name`, `email`, `password`, `role`.
2. Backend hashes password using **Bcrypt** and stores in `users` collection.
3. If role is `member` or `patient`, backend auto-generates a **6-digit numeric invite code** using `random.choices(string.digits, k=6)`.
4. Backend returns a **JWT token** (expires in 30 days, HS256 algorithm) + user object.
5. Frontend stores `authToken` and `user` in **AsyncStorage** using AuthContext.
6. Every subsequent API call attaches: `Authorization: Bearer <token>` via Axios request interceptor.
7. On app re-open, AuthContext reads from AsyncStorage to restore session automatically.
8. Logout removes both `authToken` and `user` from AsyncStorage and clears state.

**User object shape (TypeScript interface):**
```typescript
interface User {
    id: string;
    name: string;
    email: string;
    role: 'patient' | 'caregiver' | 'member' | 'guardian';
    inviteCode?: string;  // only present for patient/member role
}
```

## 5. Medicine Management

**File:** `backend/server.py`, `frontend/app/patient/medicines.tsx`

### Medicine Data Model (MongoDB document):
```
{
    _id: ObjectId,
    patientId: string,        // user's _id
    name: string,             // e.g. "Aspirin"
    dosage: string,           // e.g. "500mg"
    reminderTimes: [string],  // 24h strings, e.g. ["08:00", "20:00"]
    startDate: string,        // "YYYY-MM-DD" or null
    endDate: string,          // "YYYY-MM-DD" or null
    notes: string,            // optional free text
    createdAt: datetime
}
```

### API Endpoints:
| Method | Endpoint | Action |
|---|---|---|
| POST | `/api/medicines?user_id={id}` | Create medicine |
| GET | `/api/medicines/my?user_id={id}` | Get all medicines for user |
| PUT | `/api/medicines/{medicine_id}?user_id={id}` | Update medicine |
| DELETE | `/api/medicines/{medicine_id}?user_id={id}` | Delete medicine + all its logs |

### UI flow (medicines.tsx):
- Times are picked using **native DateTimePicker** in time mode.
- Times are stored internally as 24h `HH:MM` strings.
- Times are displayed as 12h AM/PM in the UI using `formatDisplayTime()`.
- Default new medicine: startDate = today, endDate = today + 30 days, reminderTimes = `["08:00"]`.
- Multiple reminder times supported — user can add/remove slots.
- On Save: calls API → then calls `scheduleMedicineNotifications()`.
- On Delete: calls API → then calls `cancelMedicineNotifications()`.

## 6. Notification System

**File:** `frontend/utils/notifications.ts`

### Channel (Android only):
- Channel ID: `medication-reminders`
- Importance: `AndroidImportance.MAX`
- Vibration: `[0, 250, 250, 250]`
- Light Color: `#6366F1`
- Sound: default

### Notification Content:
- Title: `"Time for your medicine"`
- Body: `"{medicineName} ({dosage})"`
- Data payload: `{ medicineId, medicineName, dosage }`
- Priority: `AndroidNotificationPriority.MAX`

### Schedule Logic:
- Trigger type: `DAILY` (repeats every day at same hour/minute)
- Uses `preciseSchedules: true`
- Unique identifier per dose: `{medicineId}_{HHmm}` (e.g. `abc123_0800`)
- If `endDate` is already past today → notifications are NOT scheduled (skipped silently)
- Before re-scheduling, old notifications for that medicine are cancelled first

### Cancel Logic:
- `cancelMedicineNotifications(medicineId)`: cancels all scheduled notifications whose identifier starts with `{medicineId}_`
- `cancelAllNotifications()`: cancels everything

## 7. Dashboard & Dose Status Logic

**File:** `backend/server.py` → `get_dashboard()` and `compute_status()`

### How the Dashboard is Built (Server-Side):
1. Frontend calls `GET /api/dashboard/{patientId}?tz_offset={offset}` where `tz_offset` = `new Date().getTimezoneOffset()` (e.g. -330 for IST).
2. Backend computes `local_now = UTC_now - timedelta(minutes=tz_offset)`.
3. Backend fetches all medicines for the patient.
4. For each medicine, checks Treatment Window:
   - If `today < startDate` → **skip** (don't show)
   - If `today > endDate` → **skip** (don't show)
5. For each active medicine, iterates over `reminderTimes`.
6. Calls `compute_status(log, scheduled_hour, scheduled_minute, local_now)`:

### `compute_status()` Rules (exact logic):
```
GRACE_PERIOD = 60 minutes

if log exists AND log.status == "taken"  → return "taken"
if local_now <= scheduledDateTime + 60min → return "pending"
else                                       → return "missed"
```

7. If status is "missed" and no log exists (or log not already finalized), the backend **auto-writes** a `missed` log to the `logs` collection via `upsert`.
8. Response item shape:
```json
{
    "id": "{medicineId}_{date}_{time}",
    "medicineId": "...",
    "name": "Aspirin",
    "dosage": "500mg",
    "scheduledTime": "08:00",
    "date": "2026-04-03",
    "status": "taken" | "pending" | "missed",
    "takenAt": "2026-04-03T07:58:00" | null
}
```
9. Items are sorted by `scheduledTime` ascending.
10. Response also includes `patient_name` field (fetched from users collection).

### Mark as Taken:
- Frontend calls `POST /api/logs/mark-taken?user_id={id}` with `{ medicineId, date, scheduledTime }`.
- Backend upserts a log with `status: "taken"`, `takenAt: datetime.utcnow()`.

## 8. History / Log System

**Endpoint:** `GET /api/logs/history/{patient_id}?days=7&tz_offset={offset}`

- Returns logs for the past N days (default 7).
- Enriches each log with `medicineName` and `dosage` from the medicines collection.
- Sorted by date descending.

## 9. Guardian–Patient Linking System

**Files:** `backend/server.py`, `frontend/app/caregiver/index.tsx`

### GuardianLink Document (MongoDB):
```
{
    _id: ObjectId,
    guardianId: string,  // guardian user's _id
    memberId: string,    // patient user's _id
    linkedAt: datetime
}
```

### API Endpoints:
| Method | Endpoint | Action |
|---|---|---|
| POST | `/api/guardian/link?user_id={guardianId}` | Link using invite code |
| DELETE | `/api/guardian/unlink/{memberId}?user_id={guardianId}` | Unlink a member |
| GET | `/api/guardian/members?user_id={guardianId}` | Get all linked members |
| GET | `/api/guardian/member/{memberId}/dashboard?user_id={guardianId}&tz_offset={offset}` | Get member's dashboard (read-only) |

### Linking Flow:
1. Patient sees their `inviteCode` (6 digits) in their Settings/Profile screen.
2. Guardian opens their app → taps `+` FAB → enters the 6-digit code in a modal.
3. `POST /api/guardian/link` → server finds the User with that inviteCode and role=member/patient.
4. Server checks no duplicate link already exists.
5. Inserts a GuardianLink document.

### Guardian Dashboard (caregiver/index.tsx):
- Lists all linked members by name + email as cards.
- Tapping a card navigates to `/caregiver/member/{memberId}`.
- Long-pressing (trash icon) triggers unlink with confirmation alert.
- FAB color: `#10B981` (Emerald Green).

### Member Detail Screen (caregiver/member/[id].tsx):
- Calls `guardianAPI.getMemberDashboard(guardianId, memberId)`.
- Shows a stats card: Total Meds, Adherence %, Taken count.
- Adherence = `(taken / total) * 100` rounded to nearest integer.
- Lists all dose cards with status badges:
  - `pending` → Yellow `#F59E0B`
  - `taken` → Green `#10B981`
  - `missed` → Red `#EF4444`
- Navigation title updates dynamically to `"{PatientName}'s Schedule"` using `res.patient_name`.
- Guardian has **read-only** access — no "Mark as Taken" button exists on this screen.

## 14. Summary for Quick Reference

> **HealthSync is a React Native + Expo mobile app written in TypeScript. It has two roles: Patient and Guardian. The backend is a Python FastAPI server. The database is MongoDB Atlas with 4 collections: users, medicines, logs, guardian_links. All time logic (missed/pending/taken) is calculated server-side using a 60-minute grace period. Notifications are scheduled locally on the device using expo-notifications with DAILY repeat triggers and MAX priority. Guardians link to patients using a 6-digit invite code. The Guardian's dashboard is read-only.**
